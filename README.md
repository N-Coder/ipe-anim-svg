# ipe-svg

Convert [Ipe](https://ipe.otfried.org/) presentations to interactive and animated,
statically-hostable [reveal.js](https://revealjs.com/) websites.

Each Ipe page becomes a reveal.js slide. Views within a page (the incremental build-up of layers) become fragment steps.
The resulting site requires no specialized software — just a browser.
See the `example/` folder for a demo that is also available live on GitHub Pages [here](https://n-coder.github.io/ipe-anim-svg/example/presentation.html).

---

## Quick start

Dependencies:
- a C++ compiler and CMake
- development versions of the Cairo and FreeType libraries
- an installed ipelib that matches version checked out in the Ipe git submodule

```bash
# 1. Build the C++ renderer
git submodule update --init
cmake -B build
cmake --build build

# 2a. Multi-file mode (needs a local HTTP server to serve the SVG files)
python3 ipe2reveal.py talk.ipe output/
cd output && python3 -m http.server
# → http://localhost:8000/presentation.html

# 2b. Standalone mode (single file, works directly from the filesystem)
python3 ipe2reveal.py talk.ipe talk.html --standalone
# → open talk.html directly in a browser
```

> **Multi-file vs standalone:**
> Multi-file mode writes one `.svg` per page plus `ipe-meta.json` and
> `ipe-layers.js` alongside `presentation.html`.  The plugin fetches those
> files at runtime, so a local HTTP server (or any static host) is required —
> browsers block `fetch()` on `file://` URLs.
>
> Standalone mode (`--standalone`) bundles everything into a single `.html`
> file: SVGs are embedded directly in the `<section>` elements and
> `ipe-layers.js` is inlined in a `<script>` block.  No server is needed; the
> file opens from the filesystem with `file://`.

## Hosting on GitHub Pages

`output/` is a static site and can be hosted anywhere. GitHub Pages is the
easiest zero-cost option:

1. Create a repository (or use an existing one).

2. Copy the contents of `output/` into a directory in the repository, for
   example `docs/`:
   ```bash
   cp -r output/ docs/
   git add docs/
   git commit -m "add presentation"
   git push
   ```

3. In the repository settings → **Pages**, set the source branch to `main`
   and the folder to `/docs`.

4. GitHub will publish the site at
   `https://<user>.github.io/<repo>/presentation.html` within a minute or two.

Alternatively, push `output/` to a dedicated `gh-pages` branch:
```bash
git subtree push --prefix output origin gh-pages
```

## How the generated presentation works

### Multi-file mode

`ipe2reveal.py` writes four kinds of files into the output directory:

| File | Purpose |
|---|---|
| `presentation.html` | User-editable skeleton — controls slide order and overrides |
| `page-01.svg`, `page-02.svg`, … | Per-page SVGs, one per Ipe page |
| `ipe-meta.json` | Page/layer/view metadata fetched by the plugin at runtime |
| `ipe-layers.js` | reveal.js plugin that drives everything |

### Standalone mode (`--standalone`)

Everything is bundled into a single `.html` file.  No separate SVG files,
`ipe-meta.json`, or `ipe-layers.js` are written.

### What the plugin does at load time

1. Reads page/layer/view metadata — from an inline `#ipe-meta` `<script>`
   block in standalone mode, or by fetching `ipe-meta.json` in multi-file mode.
2. Matches each `<section data-ipe-page="…">` to an Ipe page (see *Editing the HTML* below).
3. Appends a new `<section>` for every Ipe page not yet claimed by any existing section.
4. Injects the SVG into each section — in multi-file mode by fetching
   `page-N.svg`; in standalone mode the SVGs are already embedded so this step
   is skipped.
5. Appends any missing `.fragment.ipe-view` spans for views that were not
   hand-authored.
6. Applies the initial layer visibility (view 1) to every slide.

During navigation it listens to reveal.js events:

- **`slidechanged`** — resets the current slide to view 1.
- **`fragmentshown`** — reads `data-visible-layers` from the fragment and
  shows/hides layers accordingly.
- **`fragmenthidden`** — finds the last still-active fragment and reverts to
  its layer state (or to view 1 if none remain).

Layer transitions are pure CSS (`opacity 0.3s ease`), applied via the
`[data-ipe-layer]` attribute that the plugin stamps onto every layer element.

## Editing `presentation.html`

The skeleton looks like this:

```html
<div class="slides">
  <section data-ipe-page="1"></section>  <!-- Introduction -->
  <section data-ipe-page="2"></section>  <!-- Method -->
  <section data-ipe-page="3"></section>  <!-- Results -->
</div>
```

Everything between `<div class="slides">` and its closing tag is yours to edit.

In standalone mode the file also contains an `#ipe-meta` block and an inlined
`ipe-layers.js` `<script>` — those are generated by `ipe2reveal.py` and should
not be hand-edited.

`ipe2reveal.py` **never overwrites an existing `presentation.html`**. If the
file already exists the script prints a diff of what would change (slide
structure and, for standalone mode, the embedded metadata) and exits without
writing. Delete the file and re-run to regenerate it.

### Selecting pages

`data-ipe-page` accepts a **1-based index**, a **page title** exactly as it
appears in the Ipe document, or the special value **`auto`**:

```html
<section data-ipe-page="2"></section>       <!-- explicit index -->
<section data-ipe-page="Method"></section>  <!-- explicit title -->
<section data-ipe-page="auto"></section>    <!-- next unclaimed page -->
```

`data-ipe-page="auto"` assigns the next page that has not yet been claimed by
any other section. Resolution happens in document order, so earlier sections
claim their pages first:

```html
<!-- Three auto-sections: get pages 1, 2, 3 in order -->
<section data-ipe-page="auto"></section>
<section data-ipe-page="auto"></section>
<section data-ipe-page="auto"></section>

<!-- Mixed: explicit page 3 is claimed first, so the auto-section gets page 1 -->
<section data-ipe-page="3"></section>
<section data-ipe-page="auto"></section>
```

A `<section>` with no `data-ipe-page` at all is treated as a plain reveal.js
slide and is not touched by the plugin.

### Reordering, skipping, and auto-appending pages

Because sections are resolved at runtime you can freely reorder them, skip
pages, or show the same page more than once:

```html
<section data-ipe-page="3"></section>  <!-- show Results first -->
<section data-ipe-page="1"></section>  <!-- then Introduction -->
                                       <!-- page 2 not mentioned yet -->
```

By default the plugin appends a new `<section>` for every page that was not
claimed by any section in the HTML, so page 2 would still appear at the end.
To prevent this and show only the pages you explicitly list, add
`data-ipe-no-auto-pages` to the root `<div class="slides">`:

```html
<div class="slides" data-ipe-no-auto-pages>
  <section data-ipe-page="3"></section>  <!-- Results -->
  <section data-ipe-page="1"></section>  <!-- Introduction -->
                                         <!-- page 2 is omitted entirely -->
</div>
```

### Hand-authoring fragment steps

The plugin auto-fills the fragment spans that correspond to view steps, but you
can write them yourself if you want to customise the visible layers:

```html
<section data-ipe-page="2">
  <!-- Only show layers "intro" and "detail" in step 2, skipping "extra" -->
  <span class="fragment ipe-view" data-visible-layers="intro detail"></span>
</section>
```

`data-visible-layers` is a **space-separated list** of layer names. The plugin
appends any remaining auto-generated views as spans after your hand-authored ones.

To suppress auto-filling entirely for a section, add `data-ipe-no-auto-views`:

```html
<section data-ipe-page="2" data-ipe-no-auto-views>
  <!-- Exactly the views you write here; nothing added automatically -->
  <span class="fragment ipe-view" data-visible-layers="intro"></span>
</section>
```

To suppress view-fragment auto-fill for all slides globally, add
`data-ipe-no-auto-views` to the root `<div class="slides">`:

```html
<div class="slides" data-ipe-no-auto-views>
  <!-- No view-fragment spans will be auto-generated for any slide -->
  <section data-ipe-page="1"></section>
  <section data-ipe-page="2"></section>
</div>
```

Both attributes may be combined if you want to suppress both auto-page
assignment and auto-view generation:

```html
<div class="slides" data-ipe-no-auto-pages data-ipe-no-auto-views>
  <section data-ipe-page="1"></section>
</div>
```

### Speaker notes

The reveal.js [Notes plugin](https://revealjs.com/speaker-view/) is included automatically. Notes written in Ipe (**Edit → Notes…** for the current page) are imported by `ipe2reveal.py` and placed in the generated `presentation.html` as `<aside class="notes">` elements automatically.

Press **S** while viewing the presentation to open the speaker view in a separate window. It shows the current slide, the next slide, your notes, and a timer.

> **Tip:** The speaker window must be opened from the same browser that is driving the presentation. Both windows communicate via `localStorage`, so they need to share the same origin.

You can also add or edit notes directly in `presentation.html`:

```html
<section data-ipe-page="2">
  <aside class="notes">Mention the experimental setup here.</aside>
</section>
```

### Changing the theme and transition

Edit the reveal.js initialisation block near the bottom of `presentation.html`:

```js
Reveal.initialize({
  hash: true,
  transition: 'slide',  // none | fade | slide | convex | concave | zoom
  plugins: [RevealNotes, IpeLayers],
});
```

Swap the theme stylesheet (`white.css`) for any of the built-in reveal.js
themes (`black`, `league`, `moon`, `serif`, `sky`, `solarized`, …).

## Tools reference

### `ipe2reveal.py` — main pipeline (recommended)

Runs the full conversion in one step.

```
python3 ipe2reveal.py <input.ipe> <output_dir> [--ipe-svg PATH]
python3 ipe2reveal.py <input.ipe> <output.html> --standalone [--ipe-svg PATH]
```

| Argument | Default | Description |
|---|---|---|
| `input.ipe` | — | Ipe source file |
| `output_dir` | — | Output directory (multi-file) or `.html` file path (`--standalone`) |
| `--ipe-svg PATH` | auto-detected | Path to the `ipe_svg` binary |
| `--standalone` | off | Bundle everything into a single self-contained HTML file |

Auto-detection checks `./build/ipe_svg` and `./cmake-build-debug/ipe_svg`
relative to the script.

Running the script again regenerates the SVGs and `ipe-meta.json` (multi-file)
or the embedded SVG/metadata (standalone). `presentation.html` is **never
overwritten** — if it already exists, the script prints a diff and exits.
Delete the file and re-run to regenerate it.

---

### `ipe_svg` — C++ renderer (low-level)

Renders an Ipe document to SVG via Cairo and writes per-object class tags to
`stdout`.

```
./build/ipe_svg <input.ipe> <output.svg> [--page N]
```

Without `--page` all pages are rendered into a single SVG. With `--page N`
(1-based) only that page is rendered, which is what `ipe2reveal.py` uses
internally.

**Tag format** — one line per rendered object, space-separated tokens:

| Tag | Meaning |
|---|---|
| `objnr-X` | Object index on the page |
| `layernr-X` | Layer index |
| `layer-NAME` | Layer name |
| `page-X` | 0-based page number in the document |
| `type-TYPE` | Object type: `path`, `text`, `group`, `image`, `reference` |
| `custom-DATA` | Value of the Ipe object's custom data field |

---

### `apply_classes.py` — SVG post-processor (legacy)

Applies class tags to a rendered SVG and regroups objects by page. Accepts
both single-page and multi-page SVGs (the output of `ipe_svg` without
`--page`).

```
python3 apply_classes.py <input.svg> <tags.txt> <output.svg>
```

This step is performed automatically by `ipe2reveal.py`. Use it directly only
if you need a multi-page SVG with class annotations for a custom workflow.

---

### `ipe_info.py` — metadata extractor (legacy)

Extracts page/layer/view structure from an Ipe file and prints JSON to
`stdout`.

```
python3 ipe_info.py <input.ipe> > info.json
```

This step is performed automatically by `ipe2reveal.py`. Use it directly only
if you need the raw metadata for a custom workflow.
