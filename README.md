# ipe-svg

Convert [Ipe](https://ipe.otfried.org/) presentations to interactive and animated,
statically-hostable [reveal.js](https://revealjs.com/) websites.

Each Ipe page becomes a reveal.js slide. Views within a page (the incremental build-up of layers) become fragment steps.
The resulting site requires no specialized software — just a browser.
See the `example/` folder for a demo that is also available live on GitHub Pages [here](https://n-coder.github.io/ipe-anim-svg/example/presentation.html.

---

## Quick start

Dependencies: a C++ compiler and CMake, Cairo and FreeType development libraries (and the Ipe git submodule).

```bash
# 1. Build the C++ renderer
git submodule update --init
cmake -B build
cmake --build build

# 2. Convert your presentation
python3 ipe2reveal.py talk.ipe output/

# 3. Open in a browser (needs a local HTTP server because of SVG fetches)
cd output && python3 -m http.server
```

Then visit `http://localhost:8000/presentation.html`.

> **Why a local server?** The plugin fetches per-page SVG files at runtime.
> Browsers block `fetch()` on `file://` URLs, so a minimal HTTP server is
> required even for local viewing. `python3 -m http.server` is sufficient.

## Hosting on GitHub Pages

`output/` is a static site and can be hosted anywhere. GitHub Pages is the
easiest zero-cost option:

1. Create a repository (or use an existing one).

2. Copy the contents of `output/` into a directory in the repository, for
   example `docs/`:
   ```bash
   cp -r output/ docs/
   git add docs/
   git commit -m "add presentation"
   git push
   ```

3. In the repository settings → **Pages**, set the source branch to `main`
   and the folder to `/docs`.

4. GitHub will publish the site at
   `https://<user>.github.io/<repo>/presentation.html` within a minute or two.

Alternatively, push `output/` to a dedicated `gh-pages` branch:
```bash
git subtree push --prefix output origin gh-pages
```

## How the generated presentation works

`ipe2reveal.py` writes four kinds of files into the output directory:

| File | Purpose |
|---|---|
| `presentation.html` | User-editable skeleton — controls slide order and overrides |
| `page-01.svg`, `page-02.svg`, … | Per-page SVGs, one per Ipe page |
| `ipe-meta.json` | Page/layer/view metadata consumed by the plugin |
| `ipe-layers.js` | reveal.js plugin that drives everything |

### What the plugin does at load time

1. Reads `#ipe-meta` (embedded JSON in `presentation.html`).
2. Matches each `<section>` to an Ipe page (see *Editing the HTML* below).
3. Fetches the matching `page-N.svg` and injects it into the section.
4. Appends any missing `.fragment.ipe-view` spans for views that were not
   hand-authored.
5. Applies the initial layer visibility (view 1) to every slide.

During navigation it listens to reveal.js events:

- **`slidechanged`** — resets the current slide to view 1.
- **`fragmentshown`** — reads `data-visible-layers` from the fragment and
  shows/hides layers accordingly.
- **`fragmenthidden`** — finds the last still-active fragment and reverts to
  its layer state (or to view 1 if none remain).

Layer transitions are pure CSS (`opacity 0.3s ease`), applied via the
`[data-ipe-layer]` attribute that the plugin stamps onto every layer element.

## Editing `presentation.html`

The skeleton looks like this:

```html
<div class="slides">
  <section data-ipe-page="1"></section>  <!-- Introduction -->
  <section data-ipe-page="2"></section>  <!-- Method -->
  <section data-ipe-page="3"></section>  <!-- Results -->
</div>
```

Everything between `<div class="slides">` and its closing tag is yours to edit.
The `#ipe-meta` block and the `<script>` tags below it are regenerated by
`ipe2reveal.py` and should not be hand-edited (changes will be overwritten on
the next run).

### Selecting pages

`data-ipe-page` accepts a **1-based index** or a **page title** exactly as it
appears in the Ipe document:

```html
<section data-ipe-page="2"></section>
<section data-ipe-page="Method"></section>
```

Omit the attribute entirely to use the next not-yet-displayed page in document
order:

```html
<section></section>  <!-- first unassigned page -->
<section></section>  <!-- second unassigned page -->
```

### Reordering and skipping pages

Because sections are resolved at runtime you can freely reorder them, skip
pages, or show the same page more than once:

```html
<section data-ipe-page="3"></section>  <!-- show Results first -->
<section data-ipe-page="1"></section>  <!-- then Introduction -->
<!-- page 2 is omitted entirely -->
```

### Hand-authoring fragment steps

The plugin auto-fills the fragment spans that correspond to view steps, but you
can write them yourself if you want to customise the visible layers:

```html
<section data-ipe-page="2">
  <!-- Only show layers "intro" and "detail" in step 2, skipping "extra" -->
  <span class="fragment ipe-view" data-visible-layers="intro detail"></span>
</section>
```

`data-visible-layers` is a **space-separated list** of layer names. The plugin
appends any remaining auto-generated spans after your hand-authored ones.

To suppress auto-filling entirely for a section, add `data-ipe-no-auto`:

```html
<section data-ipe-page="2" data-ipe-no-auto>
  <!-- Exactly the spans you write here; nothing added automatically -->
  <span class="fragment ipe-view" data-visible-layers="intro"></span>
</section>
```

Apply `data-ipe-no-auto` to `<div class="slides">` to disable auto-filling
globally.

### Speaker notes

Standard reveal.js `<aside class="notes">` works as usual:

```html
<section data-ipe-page="2">
  <aside class="notes">Mention the experimental setup here.</aside>
</section>
```

### Changing the theme and transition

Edit the reveal.js initialisation block near the bottom of `presentation.html`:

```js
Reveal.initialize({
  hash: true,
  transition: 'slide',  // none | fade | slide | convex | concave | zoom
  plugins: [IpeLayers],
});
```

Swap the theme stylesheet (`white.css`) for any of the built-in reveal.js
themes (`black`, `league`, `moon`, `serif`, `sky`, `solarized`, …).

## Tools reference

### `ipe2reveal.py` — main pipeline (recommended)

Runs the full conversion in one step.

```
python3 ipe2reveal.py <input.ipe> <output_dir> [--ipe-svg PATH]
```

| Argument | Default | Description |
|---|---|---|
| `input.ipe` | — | Ipe source file |
| `output_dir` | — | Directory to write output into (created if absent) |
| `--ipe-svg PATH` | auto-detected | Path to the `ipe_svg` binary |

Auto-detection checks `./build/ipe_svg` and `./cmake-build-debug/ipe_svg`
relative to the script.

Running the script again regenerates the SVGs and `ipe-meta.json` but
**overwrites `presentation.html`**. Copy your edited skeleton somewhere safe
before re-running if you have hand-edits you want to keep, then merge them
back.

---

### `ipe_svg` — C++ renderer (low-level)

Renders an Ipe document to SVG via Cairo and writes per-object class tags to
`stdout`.

```
./build/ipe_svg <input.ipe> <output.svg> [--page N]
```

Without `--page` all pages are rendered into a single SVG. With `--page N`
(1-based) only that page is rendered, which is what `ipe2reveal.py` uses
internally.

**Tag format** — one line per rendered object, space-separated tokens:

| Tag | Meaning |
|---|---|
| `objnr-X` | Object index on the page |
| `layernr-X` | Layer index |
| `layer-NAME` | Layer name |
| `page-X` | 0-based page number in the document |
| `type-TYPE` | Object type: `path`, `text`, `group`, `image`, `reference` |
| `custom-DATA` | Value of the Ipe object's custom data field |

---

### `apply_classes.py` — SVG post-processor (legacy)

Applies class tags to a rendered SVG and regroups objects by page. Accepts
both single-page and multi-page SVGs (the output of `ipe_svg` without
`--page`).

```
python3 apply_classes.py <input.svg> <tags.txt> <output.svg>
```

This step is performed automatically by `ipe2reveal.py`. Use it directly only
if you need a multi-page SVG with class annotations for a custom workflow.

---

### `ipe_info.py` — metadata extractor (legacy)

Extracts page/layer/view structure from an Ipe file and prints JSON to
`stdout`.

```
python3 ipe_info.py <input.ipe> > info.json
```

This step is performed automatically by `ipe2reveal.py`. Use it directly only
if you need the raw metadata for a custom workflow.
